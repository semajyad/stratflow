# StratFlow V2 Features

This document catalogues all features built across Phases 1–5 of the V2 development cycle. V2 substantially extended the V1 foundation (document upload, diagram generation, work item generation) with a full prioritisation, risk, delivery, governance, and admin surface.

---

## Phase 1 — Core Delivery Pipeline

### Prioritisation (RICE / WSJF Frameworks)

- Users select either RICE or WSJF as the scoring framework for a project (`projects.selected_framework`).
- Work items can be scored individually or via AI-generated baseline scores for all items at once.
- Computed `final_score` is stored on each work item. Items can be re-ranked by score with one click.
- **Routes:** `/app/prioritisation/*`
- **Model columns added:** `rice_reach`, `rice_impact`, `rice_confidence`, `rice_effort`, `wsjf_business_value`, `wsjf_time_criticality`, `wsjf_risk_reduction`, `wsjf_job_size`, `final_score` on `hl_work_items`
- **Prompt:** `PrioritisationPrompt::RICE_PROMPT`, `PrioritisationPrompt::WSJF_PROMPT`

### Risk Modelling with AI Generation and 5x5 Heatmap

- Risks are identified manually or auto-generated by AI from the work item backlog.
- Each risk carries a 1–5 likelihood and 1–5 impact score; `priority = likelihood × impact` (1–25 scale).
- Risks are rendered on a 5x5 colour-coded heatmap in the UI.
- AI generates targeted mitigation strategies per risk, given full context.
- Risks can be linked to multiple work items via `risk_item_links`.
- **Routes:** `/app/risks/*`
- **Tables:** `risks`, `risk_item_links`
- **Prompts:** `RiskPrompt::GENERATE_PROMPT`, `RiskPrompt::MITIGATION_PROMPT`

### User Story Decomposition with Dependency Tracking

- HL Work Items are decomposed into 5–10 granular user stories via AI.
- Stories can be created manually, reordered by drag-and-drop, and exported as CSV, JSON, or Jira format.
- Each story can be sized manually or via AI suggestion (modified Fibonacci scale: 1–20 points).
- Stories support a self-referential `blocked_by` field for intra-story dependencies.
- **Routes:** `/app/user-stories/*`
- **Table:** `user_stories`
- **Prompts:** `UserStoryPrompt::DECOMPOSE_PROMPT`, `UserStoryPrompt::SIZE_PROMPT`

### Sprint Allocation with Capacity Management

- Sprints are created with a date range and team capacity in story points.
- Stories are assigned to sprints manually (drag-and-drop) or via AI auto-allocation.
- AI allocation respects story priority, sprint capacity, and `blocked_by` dependencies.
- **Routes:** `/app/sprints/*`
- **Tables:** `sprints`, `sprint_stories`
- **Prompt:** `SprintPrompt::ALLOCATE_PROMPT`

## Phase 2 — Admin Area

### User Management

- Org admins can view, create, update, and soft-delete users within their organisation.
- User seat limits are enforced against the active subscription (`subscriptions.user_seat_limit`).
- Users cannot self-delete.
- **Routes:** `/app/admin/users/*`

### Team Management

- Admins can create and manage named teams with sprint capacity values.
- Users can be added to or removed from teams; a user may belong to multiple teams.
- **Routes:** `/app/admin/teams/*`
- **Tables:** `teams`, `team_members`

### Organisation Settings

- Admins can configure org-level defaults stored in `organisations.settings_json`:
  - Custom AI persona prompts
  - HL Work Item default values
  - Capacity tripwire thresholds for the Drift Engine
- **Route:** `/app/admin/settings`

## Phase 3 — Sounding Boards

### Persona-based AI Evaluation Panels

- Projects can be evaluated by panels of named AI personas (e.g. CEO, Senior Developer, CFO).
- Each panel has a type (`executive` or `product_management`) and a list of persona members with role titles and prompt descriptions.
- Panels may be system-wide defaults (`org_id IS NULL`, managed by superadmin) or org-specific overrides.
- **Tables:** `persona_panels`, `persona_members`

### Evaluation Flow

- Evaluations are triggered per screen (work items, risks, user stories, etc.) at one of three criticism levels: `devils_advocate`, `red_teaming`, or `gordon_ramsay`.
- `SoundingBoardService` calls Gemini once per persona, collecting structured feedback (Overall Assessment, Key Concerns, Recommendations, Risk Rating).
- Results are stored as JSON in `evaluation_results` with a status of `pending`, `accepted`, `rejected`, or `partial`.
- Users can accept or reject individual persona responses.
- Evaluation history is available per project.
- **Routes:** `/app/sounding-board/*`
- **Table:** `evaluation_results`
- **Service:** `SoundingBoardService`
- **Prompt:** `PersonaPrompt::buildPrompt()`

## Phase 3 — Superadmin Area

### Organisation Management

- Superadmins can view all organisations with status, user count, and subscription info.
- Actions: suspend, enable, or delete an organisation; export all org data as JSON.
- **Routes:** `/superadmin/organisations/*`

### Default Persona Management

- Superadmins manage the system-default persona panels visible to all organisations.
- Default panels are seeded on first use if none exist.
- **Route:** `/superadmin/personas`

### Role Assignment

- Superadmins can promote any user to the superadmin role.
- **Route:** `/superadmin/assign-superadmin`

## Phase 4 — Strategic Drift Engine

### Baselines

- Users create point-in-time strategic baseline snapshots of a project's work items and story metrics.
- Snapshots are stored as JSON in `strategic_baselines` and used as the reference for drift detection.
- **Route:** `POST /app/governance/baseline`
- **Model:** `StrategicBaseline`

### Tripwires and Drift Alerts

- The Drift Engine compares the current project state against the latest baseline.
- Two automated tripwires:
  - **Capacity tripwire:** raises a `capacity_tripwire` alert when story size for a parent work item grows beyond the configured threshold.
  - **Dependency tripwire:** raises a `dependency_tripwire` alert when cross-team story blockers are detected.
- Optional AI alignment check via `DriftPrompt::ALIGNMENT_PROMPT` — raises an `alignment` alert when new stories do not serve the original OKRs.
- **Route:** `POST /app/governance/detect`
- **Tables:** `drift_alerts`, `governance_queue`
- **Models:** `DriftAlert`, `GovernanceItem`
- **Service:** `DriftDetectionService`

### Governance Dashboard

- Lists active drift alerts, pending governance queue items, and baseline history for a project.
- Alerts can be acknowledged or resolved; governance items can be approved or rejected.
- Approving a governance item clears `requires_review` on the related work item or user story.
- **Routes:** `/app/governance/*`

### `requires_review` Flag

- `hl_work_items.requires_review` and `user_stories.requires_review` are set to `1` when the Drift Engine flags a change.
- Cleared when the related governance queue item is approved.

## Phase 5 — V2 Completion

### HL Work Item Dependencies

- Work items can have blocking dependencies on other work items in the same project.
- A row `(item_id, depends_on_id)` means `depends_on_id` must be completed before `item_id` begins.
- Dependencies are managed via `createBatch()` — replaces the full set for an item atomically.
- Self-references and zero IDs are silently skipped.
- `ON DUPLICATE KEY UPDATE` prevents errors on re-save.
- **Table:** `hl_item_dependencies`
- **Model:** `HLItemDependency`
- **Used by:** `WorkItemController`, `DriftDetectionService::checkDependencyTripwire()`

### Manual Work Item Creation

- Users can create HL Work Items manually without running AI generation.
- **Route:** `POST /app/work-items/store`

### Regenerate Sizing — Work Items

- Re-runs AI sprint estimation for all work items in a project in a single call.
- Uses `WorkItemPrompt::SIZING_PROMPT` to return a JSON array of `{id, estimated_sprints}` pairs.
- **Route:** `POST /app/work-items/regenerate-sizing`
- **Prompt:** `WorkItemPrompt::SIZING_PROMPT`

### Regenerate Sizing — User Stories

- Re-runs AI story point sizing for all user stories in a project in a single call.
- **Route:** `POST /app/user-stories/regenerate-sizing`

### Invoice Management

- Org admins can view a list of Stripe invoices for their organisation.
- Individual invoices can be downloaded as PDFs directly from the admin area.
- **Routes:** `GET /app/admin/invoices`, `GET /app/admin/invoices/{id}/download`

### Workflow Navigation Partial

- A shared `templates/partials/workflow-nav.php` component renders the step-by-step progress bar across all authenticated app screens.
- Provides a consistent visual indicator of the user's position in the strategy-to-delivery workflow.

### Stripe Product Extensions

- Subscription model extended to support additional Stripe product types:
  - **User packs:** additional seat bundles purchasable as add-ons (`subscriptions.user_seat_limit`)
  - **Evaluation boards:** grants access to the Sounding Board feature (`subscriptions.has_evaluation_board`)

### Cross-cutting UI Improvements

- Consistent error and success flash messages across all admin and app screens.
- Form validation feedback inline on all create/edit forms.
- Responsive table layouts for work items, user stories, risks, and sprint allocation screens.
