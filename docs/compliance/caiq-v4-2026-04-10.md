## Executive Summary

StratFlow is a multi-tenant SaaS built on PHP 8.4 / MySQL 8.4, deployed via cPanel-managed hosting at stratflow.threepointssolutions.com. The platform converts strategic plans into work items, user stories, and sprint plans, with integrations to Stripe (billing), Gemini (AI generation), GitHub/GitLab, Jira, and Xero. Tenant data is isolated by org_id enforced at the query layer in every model.

This document provides evidence-based answers to CAIQ v4 control domains. Each answer cites a specific file, class, or documented control. Where a control is absent or partial, it is marked honestly.

### Controls we can confirm today (Yes)

- AIS-01/02/03: PDO prepared statements throughout src/Models/, Sanitizer input cleansing, htmlspecialchars template output.
- AIS-05 baseline: CSRF middleware validates tokens on POST/PUT/DELETE in src/Middleware/CSRFMiddleware.php.
- CEK-01: HSTS max-age=31536000 with includeSubDomains and preload set in src/Core/Response.php.
- CEK-02: Passwords hashed with password_hash(PASSWORD_DEFAULT) (bcrypt) in src/Core/Auth.php.
- DSP-02: Multi-tenant isolation via org_id filter on every model query.
- IAM-02: RBAC via AuthMiddleware, AdminMiddleware, SuperadminMiddleware, ExecutiveMiddleware, BillingMiddleware.
- IAM-06: Authentication events logged through src/Services/AuditLogger.php.
- IAM-07: 30-minute session inactivity timeout with periodic ID regeneration (src/Core/Session.php).
- IAM-08: 12-char password policy with upper/lower/number/special (src/Core/PasswordPolicy.php).
- LOG-01: Security events captured in audit_log table via AuditLogger (20+ event types).
- TVM-01: composer audit scheduled weekly; current scan reports zero advisories.

### Controls in progress / Partial

## AIS -- Application & Interface Security

**[AIS-01]** Do you use secure coding standards (e.g. OWASP)?
**Answer:** Yes
**Evidence:** Development follows OWASP Top 10:2025. CLAUDE.md mandates PDO prepared statements, htmlspecialchars output encoding, CSRF middleware on state-changing POSTs, and PHP 8.4 strict types. Monthly security assessment reports in docs/compliance/ track OWASP coverage. A pre-commit security-auditor subagent runs before every commit.

**[AIS-02]** Is user input validated/sanitised before processing?
**Answer:** Yes
**Evidence:** src/Core/Sanitizer.php provides centralised input cleansing. src/Core/Request.php wraps superglobal access. Controllers perform per-field validation before persistence.

**[AIS-03]** Are SQL injection protections in place (prepared statements)?
**Answer:** Yes
**Evidence:** src/Core/Database.php exposes a single query(sql, params) helper backed by PDO::prepare with parameter binding. String concatenation of SQL is forbidden by convention. The 2026-04-10 security assessment marks Injection Prevention as PASS.

**[AIS-04]** Are XSS protections in place?
**Answer:** Partial
**Evidence:** Template output uses htmlspecialchars with ENT_QUOTES and UTF-8. Response headers include X-Frame-Options: DENY, X-Content-Type-Options: nosniff, and a CSP (src/Core/Response.php:108-116). Gap: CSP script-src includes unsafe-inline (MEDIUM-02). Remediation is scheduled.

**[AIS-05]** Is CSRF protection implemented on state-changing requests?
**Answer:** Partial
**Evidence:** src/Middleware/CSRFMiddleware.php validates the _csrf_token field on POST/PUT/DELETE and returns 403 on failure. Tokens are auto-injected into templates via Response::render. Gap: 12 authenticated POST routes omit the csrf middleware entry in src/Config/routes.php (MEDIUM-01).

**[AIS-06]** Are file uploads validated for type and stored securely?
**Answer:** Partial
**Evidence:** src/Controllers/UploadController.php records MIME type on upload and stores files outside the public web root. smalot/pdfparser handles PDF text extraction. Gap: server-side MIME sniffing via finfo_file with an allow-list should be strengthened.

## CEK -- Cryptography, Encryption & Key Management

**[CEK-01]** Is data encrypted in transit (TLS)?
**Answer:** Yes
**Evidence:** Production uses HTTPS via Lets Encrypt (docs/DEPLOYMENT.md). HSTS header set to max-age=31536000; includeSubDomains; preload in src/Core/Response.php. Session cookies set secure and httponly flags.

**[CEK-02]** Are passwords stored using a strong one-way hash (bcrypt/argon2)?
**Answer:** Yes
**Evidence:** password_hash with PASSWORD_DEFAULT (bcrypt in PHP 8.4); password_verify in src/Core/Auth.php::attempt(). No plaintext or reversible password storage.

**[CEK-03]** Are encryption keys stored separately from encrypted data?
**Answer:** Yes
**Evidence:** Secrets (DB credentials, Stripe keys, Gemini API key, Jira tokens, GitHub App private key) are loaded from .env via vlucas/phpdotenv. .env is excluded from deployment rsync per docs/DEPLOYMENT.md Step 3 and is not committed.

**[CEK-04]** Is sensitive data encrypted at rest?
**Answer:** Partial
**Evidence:** Database-level at-rest encryption depends on the MySQL configuration at the hosting provider. Application-layer field encryption is NOT implemented for PII. Stripe card data is handled entirely by Stripe Checkout. OAuth tokens for Jira/GitHub/Xero are stored in the DB without additional envelope encryption.

## IAM -- Identity & Access Management

**[IAM-01]** Is multi-factor authentication available?
**Answer:** No
**Evidence:** No TOTP, WebAuthn, or SMS 2FA code exists in src/Core/Auth.php or elsewhere. On the roadmap; no committed delivery date.

**[IAM-02]** Is role-based access control (RBAC) implemented?
**Answer:** Yes
**Evidence:** Users carry a role field and feature flags (has_billing_access, has_executive_access) loaded at login in src/Core/Auth.php::login(). Middleware guards: AuthMiddleware, AdminMiddleware, SuperadminMiddleware, ExecutiveMiddleware, BillingMiddleware.

**[IAM-03]** Are privileged accounts separately managed?
**Answer:** Yes
**Evidence:** The superadmin role is distinct from org-scoped admin. SuperadminMiddleware restricts routes; privileged actions emit AuditLogger::ADMIN_ACTION or USER_ROLE_CHANGED events.

**[IAM-04]** Are access rights reviewed periodically?
**Answer:** No
**Evidence:** No documented periodic access review process.

**[IAM-05]** Is there a process for revoking access on offboarding?
**Answer:** Partial
**Evidence:** SuperadminController and AdminController expose user deletion, terminating access immediately. No written offboarding runbook.

**[IAM-06]** Are authentication events logged?
**Answer:** Yes
**Evidence:** src/Services/AuditLogger.php defines LOGIN_SUCCESS, LOGIN_FAILURE, LOGOUT, PASSWORD_CHANGE, PASSWORD_RESET_REQUEST constants. Failed login attempts rate-limited via login_attempts table -- Auth::isRateLimited blocks at 5 failures per IP per 15 minutes.

**[IAM-07]** Is there session timeout / automatic logout?
**Answer:** Yes
**Evidence:** src/Core/Session.php enforces a 30-minute inactivity timeout, periodic session ID regeneration every 15 minutes, and session_regenerate_id(true) on login. Cookies are secure, httponly, SameSite=Lax.

**[IAM-08]** Are password complexity requirements enforced?
**Answer:** Yes
**Evidence:** src/Core/PasswordPolicy.php enforces 12-character minimum, at least one uppercase, lowercase, number, and special character. Validation runs on signup and password change.

## LOG -- Logging & Monitoring

**[LOG-01]** Are security events logged (auth failures, access control violations)?
**Answer:** Yes
**Evidence:** src/Services/AuditLogger.php logs 20+ event types (auth, user mgmt, data export, integration connect/disconnect, webhook delivery) to the audit_log table with actor, IP, user agent, and context JSON. Failed logins additionally recorded in login_attempts for rate limiting.

**[LOG-02]** Are logs stored securely and tamper-resistant?
**Answer:** Partial
**Evidence:** audit_log rows are append-only by convention but the table has no cryptographic tamper-evidence (no hash chain or WORM storage). Database access restricted to the application user.

**[LOG-03]** Are logs retained for an appropriate period?
**Answer:** Partial
**Evidence:** Audit log rows retained indefinitely in the current schema (no automated purge). No formal retention policy defines the appropriate period.

**[LOG-04]** Is there alerting on anomalous events?
**Answer:** No
**Evidence:** No automated alerting pipeline on audit events. Rate limiting on failed logins blocks brute force but does not notify operators.

**[LOG-05]** Are audit logs available to customers?
**Answer:** No
**Evidence:** The admin UI does not currently expose audit_log to org admins. Planned for a future release.
