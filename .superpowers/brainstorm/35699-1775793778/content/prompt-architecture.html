Org config injection — how custom rules reach the prompt

At generation time, the controller loads story_quality_config rows for the org (active only, ordered by display_order). These are formatted and appended to the prompt as a rules block:

--- ORG QUALITY RULES ---
Splitting patterns available: SPIDR, Happy/Unhappy Path,
  User Role, Performance Tier, CRUD, [Custom: Event Storming Boundaries]
Mandatory conditions:
  - Every story must include at least one non-functional requirement
  - Every epic must reference a specific market segment
-------------------------

Default patterns are seeded for every new org. Admins add/remove via Settings → Story Quality Rules.

Jira sync — new fields pushed
acceptance_criteria → Jira field acceptance_criteria (standard field on Jira Story/Epic issue type). If field not present on issue type, appended as a labelled section in the description.
kr_hypothesis → Jira custom field cf[kr_hypothesis] if configured, otherwise appended to description under "KR Hypothesis" heading.
