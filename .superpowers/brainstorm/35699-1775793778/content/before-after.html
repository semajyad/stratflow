Prompt Engineering — Before vs After

What AI currently generates vs what it will generate with the new prompts

BEFORE — Current output
Epic
Improve Checkout Flow

Reduce the number of steps in the checkout process to improve conversion rates. This epic covers frontend and backend changes needed to streamline the purchase journey.

No acceptance criteria. No KR linkage. No SMART objective. Vague scope.

Story 1 of 3
As a user, I want a better checkout, so that it's easier

Improve the checkout UI.

Not INVEST-compliant. No ACs. Vague value. No splitting pattern used.

Story 2 of 3
As a user, I want to save my payment details

Allow users to save card details for future purchases.

No ACs. Not estimable. No KR linkage. Not testable as written.

AFTER — New prompt output
Epic
Reduce checkout from 5 steps to 2 to increase conversion rate from 2.1% to 3.5%

Streamline the purchase journey by consolidating address, payment, and review screens into a single-page flow. Scope: guest and registered users on web only. Excludes mobile app (separate epic).

Acceptance Criteria
  • Checkout completes in ≤2 steps for registered users with saved payment
  • Guest checkout available without account creation
  • Conversion rate measurable via existing analytics within 1 sprint
  • No regression in payment success rate (currently 97.2%)

↗ Contributes to KR: Increase conversion rate to 3.5%

Story 1 — Happy path (splitting pattern: Happy/Unhappy Path)
As a registered user with saved payment, I want to complete checkout in one click, so that I can purchase without re-entering details and reduce abandoned carts
Acceptance Criteria
  • Given I have saved payment + address, when I click "Buy now", then order is placed without additional screens
  • Given order placed, when confirmed, then confirmation email sent within 30s
  • Checkout flow completes in <3s on 4G

Size: 3pts  ·  INVEST ✓  ·  ↗ KR: conversion rate

Key changes in the new prompts
INVEST
AI self-checks each story against all 6 INVEST criteria before outputting
SMART ACs
Acceptance criteria in Given/When/Then with measurable conditions
KR linkage
Each item explicitly states which KR it contributes to
Splitting patterns
AI names the pattern used — teams learn why stories are split this way
Value clarity
"So that..." ends with a measurable business outcome, not a vague benefit
Org rules
Custom conditions (e.g. NFRs) loaded from story_quality_config and injected into prompt
